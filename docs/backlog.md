# Backlog

1. Run it on real audio — this is the most important thing right now.
The whole pipeline has only been tested with synthetic WAVs (silence, sine waves, click tracks). You need to feed it actual MP3s and watch what happens. The stages most likely to surface real problems are: structure alignment (the hybrid labelling logic is the most speculative piece in the codebase), the LLM scoring prompts (the prompt_hint text is drafted but has never been iterated against actual Ollama responses), and energy's verse/chorus delta (which silently falls back to the half-split if sections aren't labelled yet). A single pass through the 1920s_1930s mode with a handful of Internet Archive recordings would stress-test all of this at once, because those tracks are the hardest case for every module.
2. Test fixtures — no actual MP3s in tests/fixtures/ yet.
SOURCES.md is a placeholder. This matters for two reasons: the integration tests currently mock _compute_* at the boundary, which means the librosa/essentia/Whisper code paths have never actually run under test. And the 1920s_1930s mode was designed specifically around these recordings being available. Sourcing 3–5 tracks from the Internet Archive 78rpm collection would give you both a real integration test and a demo dataset.
3. The JS audio player has no position sync to the energy visualisation.
The player JS in track.html is functional — play/pause, stem switching, scrubbing all work. But the design doc specifies "playback position synced to section visualisation." The energy bar chart and section timeline don't react to playhead position at all currently. This is the only designed feature that's explicitly unimplemented, and it's visible on every track page.
4. The criteria_contemporary.toml file needs verifying.
The 1988 and 1920s_1930s TOMLs are both complete and load cleanly (you can see this from the integration tests passing). The contemporary mode introduced hook_timing as a new criterion and dropped breakdown/double_chorus — it's present but has had less scrutiny than the other two modes, and the hook_timing prompt hint hasn't been tested against real LLM output.
5. tests/fixtures/SOURCES.md is empty, and there are no end-to-end integration tests.
Every test currently runs against in-memory SQLite and mocked audio. There's no test that runs the actual pipeline from MP3 to scored report, even with a very short track. A single tests/integration/test_pipeline_e2e.py that runs a 30-second public domain MP3 through all 8 stages (with real librosa, real Whisper on a short file, mocked Ollama) would catch a huge class of bugs that unit tests can't reach.
